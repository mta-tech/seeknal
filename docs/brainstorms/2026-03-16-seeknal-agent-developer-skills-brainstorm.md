# Seeknal Agent Developer Skills

## What We're Building

Extend seeknal so that **any coding agent** (Claude Code, Codex, Cursor, seeknal ask) can build complete seeknal projects — data pipelines, feature groups, ML models, semantic models, metrics, and exposures — using the existing `draft -> dry-run -> apply` workflow.

Three deliverables:

1. **Project-level CLAUDE.md + AGENTS.md** — Generated by `seeknal init`, these files teach any coding agent the seeknal conventions, CLI commands, YAML schemas, and project-specific context (connections, entities, existing nodes).

2. **Claude Code skills plugin** — A set of Claude Code skills (also usable as plain markdown guides by Codex/Cursor/etc.) that provide step-by-step guidance for building each type of seeknal artifact:
   - `/seeknal-pipeline` — Build sources + transforms end-to-end
   - `/seeknal-feature-group` — Create feature groups with entity, materialization
   - `/seeknal-model` — Train/serve ML models in the pipeline
   - `/seeknal-analytics` — Semantic models, metrics, exposures, reports
   - `/seeknal-rules` — Data quality rules and profiling
   - `/seeknal-deploy` — Environment setup, Prefect deployment, Docker

3. **Seeknal Ask write tools** — New tools for the Ask agent that use the draft workflow with confirmation prompts before applying:
   - `draft_node` — Generate YAML/Python draft for any node type
   - `dry_run_draft` — Validate a draft file
   - `apply_draft` — Apply after user confirmation
   - `run_pipeline` — Execute the pipeline
   - `edit_node` — Modify existing pipeline nodes

## Why This Approach

### Skills-First Architecture

Skills are the reusable knowledge unit because:
- **Claude Code users** get slash commands (`/seeknal-pipeline`) with rich, contextual guidance
- **Codex/Cursor/other agents** can read the same skill markdown as documentation
- **Seeknal Ask** gains write tools that follow the same patterns internally
- **Single source of truth** — skills define the canonical way to build each artifact type

### Draft Workflow as Safety Layer

All write operations go through `seeknal draft -> dry-run -> apply`:
- Drafts are temporary files that don't affect the project until applied
- `dry-run` validates YAML schema, SQL syntax, ref() references
- `apply` shows a diff and moves the file to production location
- For seeknal ask: agent shows the draft content and asks for user confirmation before applying
- For Claude Code: normal permission model applies (user approves tool calls)

### Generated CLAUDE.md per Project

`seeknal init` generates a CLAUDE.md tailored to the project because:
- Different projects have different sources (CSV, PostgreSQL, Iceberg)
- Connection profiles vary per project
- Entity definitions and existing nodes provide context
- A static CLAUDE.md can't know the project's specific state

## Key Decisions

### 1. Skills are Claude Code plugin format but markdown-portable

Skills use the `.claude/skills/SKILL.md` format for Claude Code integration. But each skill is written so that any agent reading the markdown can follow the instructions — no Claude Code-specific APIs. The skills reference CLI commands (`seeknal draft`, `seeknal apply`) that work universally.

### 2. `seeknal init` generates CLAUDE.md + AGENTS.md

When `seeknal init --name my_project` runs, it also creates:
- `CLAUDE.md` — Project conventions, available commands, YAML schema reference, current project state summary
- `AGENTS.md` — Agent-specific instructions: which tools to use, how to validate work, safety guidelines

These are regenerated/updated when running `seeknal parse` (manifest rebuild) to keep project context current.

### 3. Ask agent uses confirmation prompts for writes

The Ask agent's new write tools follow this pattern:
1. Agent generates draft content (YAML or Python)
2. Agent shows the content to the user with explanation
3. Agent asks "Apply this draft?" with confirmation prompt
4. Only on confirmation: runs `seeknal dry-run` then `seeknal apply`

This mirrors Claude Code's permission model but within the seeknal ask chat interface.

### 4. Full pipeline lifecycle in MVP

All node types are supported from the start:
- Sources (CSV, Parquet, JSONL, PostgreSQL, Iceberg, Python custom)
- Transforms (SQL, Python)
- Feature groups (with entity, materialization)
- Models (ML training/serving)
- Aggregations, second-order aggregations
- Rules (data quality checks)
- Profiles (statistical profiling)
- Semantic models, metrics
- Exposures (reports)
- Common configuration

The skills are organized per-persona (data engineer, analytics engineer, ML engineer) but cover all types.

### 5. Skills distributed as Claude Code plugin

The skills are packaged as a Claude Code plugin that users install:
```bash
claude plugins install seeknal-skills
```

Or manually by copying `.claude/skills/` into their project. This is also how Codex/Cursor users would use them — by reading the markdown files in their project.

### 6. AGENTS.md is seeknal-specific agent guidance

AGENTS.md focuses on:
- How to discover existing pipeline state (`seeknal plan`, `seeknal lineage --ascii`)
- How to validate work (`seeknal dry-run`, `seeknal run`)
- Safety rules (never modify `target/`, always use draft workflow, validate SQL)
- Project-specific context (available connections, entities, node inventory)

## Scope Boundaries

### In Scope
- CLAUDE.md + AGENTS.md generation in `seeknal init`
- Claude Code skills plugin (6-8 skills covering full lifecycle)
- Seeknal Ask write tools (draft, dry-run, apply, run, edit)
- Skills usable as plain markdown by any agent
- Confirmation prompt flow for Ask agent writes

### Out of Scope (Future)
- MCP server for universal tool access (Phase 2)
- Auto-updating CLAUDE.md on every `seeknal run` (regenerate on `seeknal parse` only)
- Visual pipeline builder UI
- Multi-agent orchestration (one agent per pipeline stage)
- Skills marketplace or registry

## Resolved Questions

1. **CLAUDE.md size strategy** — Concise + CLI discovery. CLAUDE.md covers conventions and workflows (~500 lines). Agents discover schemas via `seeknal draft <type> --help` and reading generated templates. No inline schema dumps.

2. **Skill versioning** — Always target latest seeknal. No version pinning. When seeknal updates, skills update too. Simpler maintenance.

3. **Multi-node pipeline confirmation** — Batch plan + single confirm. Agent shows the full plan (all nodes it will create), gets one confirmation, then creates all drafts, dry-runs each, and applies in order.

## Success Criteria

- A Claude Code user can run `/seeknal-pipeline` and build a working 3-node pipeline (source -> transform -> output) without reading seeknal docs
- A Codex user can read the skills markdown and achieve the same result
- Seeknal Ask can create and apply a draft node with user confirmation
- `seeknal init` generates a CLAUDE.md that makes any coding agent immediately productive
- All write operations go through the draft workflow — no direct file manipulation
