# feat: Seeknal Agent Developer Skills

## Overview

Extend seeknal so that **any coding agent** (Claude Code, Codex, Cursor, seeknal ask) can build complete seeknal projects — data pipelines, feature groups, ML models, semantic models, metrics, and exposures — using the existing `draft -> dry-run -> apply` workflow.

Three deliverables:
1. **Project-level CLAUDE.md + AGENTS.md** generated by `seeknal init`
2. **Claude Code skills** (also readable by Codex/Cursor as plain markdown)
3. **Seeknal Ask write tools** with confirmation prompts

(see brainstorm: `docs/brainstorms/2026-03-16-seeknal-agent-developer-skills-brainstorm.md`)

## Problem Statement

Today, seeknal has excellent CLI tooling (`draft -> dry-run -> apply -> plan -> run`) but **no agent-friendly interface**. A coding agent encountering a seeknal project has no way to know:

- What CLI commands are available and their exact signatures
- What YAML schemas each node type expects
- What the `draft -> dry-run -> apply` workflow is
- What connections, entities, and nodes already exist in the project

The Ask agent can analyze data (12 read-only tools) but cannot create or modify pipeline definitions. This makes seeknal invisible to the growing ecosystem of AI coding agents.

## Proposed Solution

### Skills-First Architecture

Skills are the reusable knowledge unit (see brainstorm: Key Decision #1):

```
┌─────────────────────────────────────────────────┐
│                  Seeknal Skills                  │
│  (SKILL.md files — Claude Code plugin format)    │
│  /seeknal-pipeline, /seeknal-feature-group, ...  │
├──────────────┬──────────────┬────────────────────┤
│ Claude Code  │ Codex/Cursor │   Seeknal Ask      │
│ (slash cmd)  │ (read .md)   │   (write tools)    │
├──────────────┴──────────────┴────────────────────┤
│            seeknal CLI (draft/dry-run/apply)      │
│            seeknal parse (manifest rebuild)       │
└─────────────────────────────────────────────────┘
```

- **Claude Code users** get slash commands with rich guidance
- **Codex/Cursor users** read the same SKILL.md as documentation
- **Seeknal Ask** gains write tools that follow the same draft workflow
- **CLAUDE.md + AGENTS.md** generated per project teach agents the conventions

#### Phase 2: Ask Agent Write Tools

**Goal**: Seeknal Ask can create, validate, and apply pipeline nodes with user confirmation.

**Security Model** (addresses SpecFlow Q1/Q4):
- Every write operation requires explicit user confirmation via chat
- Writable paths: `seeknal/sources/`, `seeknal/transforms/`, `seeknal/feature_groups/`, `seeknal/models/`, `seeknal/aggregations/`, `seeknal/rules/`, `seeknal/profiles/`, `seeknal/exposures/`, `seeknal/semantic_models/`, `seeknal/metrics/`, `seeknal/common/`, and project root (draft files only)
- Blocked paths: `profiles.yml`, `.env`, `target/`, `seeknal_project.yml`, `.git/`
- Add `fs_lock` to `ToolContext` for filesystem write safety (addresses SpecFlow Gap 3.11)

**Tasks**:

- [x] Add `fs_lock: threading.Lock` to `ToolContext` in `src/seeknal/ask/agents/tools/_context.py`
  - Used by all write tools to prevent concurrent filesystem modifications

- [x] Create `src/seeknal/ask/agents/tools/_write_security.py`
  - `WRITABLE_DIRS` allowlist for write operations
  - `BLOCKED_WRITE_FILES` blocklist (profiles.yml, .env, seeknal_project.yml)
  - `validate_write_path(project_path, target_path)` function
  - Reuse `_security.py` path traversal checks

- [x] Create `draft_node` tool (`src/seeknal/ask/agents/tools/draft_node.py`)
  - Parameters: `node_type: str`, `name: str`, `content: str` (YAML or Python content), `use_template: bool = False`
  - Hybrid approach (SpecFlow Q2): if `use_template=True`, calls `draft_command()` internally; if `content` provided, validates against schema and writes
  - Validates `node_type` against known types, `name` against naming rules
  - Writes draft file to project root: `draft_{type}_{name}.yml`
  - Returns: draft file path + content preview
  - Does NOT apply — just creates the draft file

- [x] Create `dry_run_draft` tool (`src/seeknal/ask/agents/tools/dry_run_draft.py`)
  - Parameters: `file_path: str` (path to draft file)
  - Calls `dry_run_command()` from `src/seeknal/workflow/dry_run.py`
  - Returns: validation result (pass/fail), errors, warnings, SQL preview if applicable
  - Read-only — no confirmation needed

- [x] Create `apply_draft` tool (`src/seeknal/ask/agents/tools/apply_draft.py`)
  - Parameters: `file_path: str`, `confirmed: bool`
  - If `confirmed=False`: returns the target path and asks agent to confirm with user
  - If `confirmed=True`: calls `apply_command()` from `src/seeknal/workflow/apply.py`
  - Handles name collisions: returns error with existing file info (SpecFlow Q6)
  - Auto-regenerates manifest after apply (SpecFlow Gap 3.12)
  - Acquires `fs_lock` during write
  - Returns: success/failure, target path, manifest diff

- [x] Create `edit_node` tool (`src/seeknal/ask/agents/tools/edit_node.py`)
  - Parameters: `node_id: str` (e.g., `source.customers`), `new_content: str`, `confirmed: bool`
  - Reads existing node file from manifest, generates diff
  - If `confirmed=False`: returns diff preview for agent to show user
  - If `confirmed=True`: writes file, runs dry-run, regenerates manifest
  - Acquires `fs_lock` during write
  - Returns: diff, validation result

- [x] Create `plan_pipeline` tool (`src/seeknal/ask/agents/tools/plan_pipeline.py`)
  - Parameters: `nodes: Optional[str]` (comma-separated node filter), `tags: Optional[str]`
  - Builds DAG via `DAGBuilder`, returns structured plan (not CLI text) (SpecFlow Q9)
  - Returns: ordered layers, node count, edge count, estimated execution order
  - Read-only — no confirmation needed

- [x] Create `show_lineage` tool (`src/seeknal/ask/agents/tools/show_lineage.py`)
  - Parameters: `node_id: Optional[str]`, `column: Optional[str]`
  - Builds fresh DAG from filesystem (SpecFlow Q9 — not cached manifest)
  - Returns ASCII tree string
  - Read-only — no confirmation needed

- [x] Create `run_pipeline` tool (`src/seeknal/ask/agents/tools/run_pipeline.py`)
  - Parameters: `nodes: Optional[str]`, `tags: Optional[str]`, `full_refresh: bool = False`, `confirmed: bool`
  - Timeout: 300 seconds default, configurable (SpecFlow Q3)
  - Runs `seeknal run` as subprocess, captures stdout/stderr
  - If `confirmed=False`: returns planned execution summary, asks agent to confirm
  - If `confirmed=True`: executes and returns result
  - Returns: exit code, output summary, node execution statuses

- [x] Register all new tools in `src/seeknal/ask/agents/agent.py` TOOLS list
  - Add to the tool list alongside existing tools
  - Update system prompt to describe write tool capabilities and confirmation protocol

  - Add batch planning guidance: "For multi-node requests, present a complete plan and get a single confirmation before proceeding." (see brainstorm: Resolved Question #3)

## Alternative Approaches Considered

### MCP-First (Rejected)

Expose seeknal as an MCP server with tools like `draft_source`, `apply_draft`, `run_pipeline`. Universal agent compatibility but adds operational complexity (server process, port management). Deferred to a future phase if demand exists.

### Ask Agent Extension Only (Rejected)

Only extend seeknal ask with write tools, no separate skills or CLAUDE.md. Simplest implementation but doesn't help Claude Code, Codex, or Cursor users — the largest and fastest-growing audience.

(see brainstorm: "Why This Approach" section for detailed rationale)

## System-Wide Impact

### Interaction Graph

- `seeknal init` → generates CLAUDE.md, AGENTS.md, `.claude/skills/` (new outputs alongside existing seeknal_project.yml, profiles.yml)
- Ask write tools → `draft_command()` → file write → `dry_run_command()` → `apply_command()` → `DAGBuilder.build()` → manifest regeneration
- Skills → user's Claude Code agent → Bash tool → seeknal CLI commands (no code-level integration)

### Error & Failure Propagation

- `draft_node` errors: invalid node type, invalid name, schema validation failure → returned as tool error message to agent → agent explains to user
- `apply_draft` errors: file collision, path validation failure, manifest parse failure → returned as tool error → agent presents options (overwrite/rename/cancel)
- `run_pipeline` errors: subprocess timeout (300s), non-zero exit code → returned with stderr → agent diagnoses
- Partial batch failure (SpecFlow Q8): no rollback. Agent reports which nodes succeeded/failed and asks how to proceed.

### State Lifecycle Risks

- Draft files in project root are temporary — `apply_draft` moves them to `seeknal/`. Orphaned drafts are possible if user abandons mid-flow. Mitigation: drafts are harmless `.yml` files that `seeknal parse` ignores (outside `seeknal/` directory).
- Manifest staleness after writes: mitigated by auto-regeneration in `apply_draft` and `edit_node`.
- Concurrent writes: mitigated by `fs_lock` in ToolContext.

### API Surface Parity

- Ask tools and Claude Code skills use the same underlying CLI commands (`seeknal draft`, `seeknal dry-run`, `seeknal apply`, `seeknal run`)
- Ask tools call Python functions directly; skills invoke CLI via Bash
- Both produce identical results because the CLI is the canonical interface

### Integration Test Scenarios

1. **Full pipeline via Ask tools**: init → draft source → dry-run → apply → draft transform → dry-run → apply → plan → run → verify output exists in `target/intermediate/`
2. **Edit existing node via Ask**: apply a source → edit_node to change SQL → dry-run → verify diff is correct → apply
3. **Batch creation via Ask**: request 3-node pipeline → agent plans all nodes → single confirmation → all drafts created and applied → plan shows correct DAG
4. **Name collision handling**: apply `source.customers` → try to apply again → agent receives collision error → presents options
5. **Claude Code skill end-to-end**: `seeknal init` → invoke `/seeknal-pipeline` → follow guided steps → verify working pipeline

## Dependencies & Prerequisites

- Seeknal `draft`, `dry-run`, `apply`, `plan`, `run`, `lineage`, `parse` commands must be stable (they are)
- LangChain `@tool` decorator pattern (already used by 12 existing tools)
- Jinja2 template engine (already used for YAML templates)
- No new pip dependencies required

## Risk Analysis & Mitigation

| Risk | Probability | Impact | Mitigation |
|------|------------|--------|------------|
| Security: prompt injection causes unwanted writes | Medium | High | `confirmed` parameter + path allowlist + fs_lock |
| CLAUDE.md becomes stale after project changes | Medium | Medium | `seeknal docs regenerate` command |
| Skills contain incorrect CLI commands after seeknal updates | Low | Medium | Always target latest; skills reference `--help` for discovery |
| `run_pipeline` timeout too short for large projects | Medium | Low | Configurable via `SEEKNAL_RUN_TIMEOUT` env var |
| Batch apply fails midway leaving partial state | Low | Medium | No rollback; agent reports partial success and asks user |
